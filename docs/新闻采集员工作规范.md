# 新闻采集员（News Collector Agent）- 工作规范文档

## 一、角色定义

新闻采集员是整个股票分析系统的**数据入口**，负责从各渠道采集原始新闻和消息，进行清洗、整理、分类后输出结构化数据，为下游Agent提供高质量的原材料。

---

## 二、核心工作职责

### 2.1 定时采集任务

| 任务类型 | 执行频率 | 说明 |
|---------|---------|------|
| 日常采集 | 每30分钟 | 抓取最新财经新闻 |
| 重要时段强化采集 | 交易时间段（9:30-11:30, 13:00-15:00）| 增加采集频率至15分钟/次 |
| 重大事件触发采集 | 事件驱动 | 监测到重大财经事件时立即触发 |
| 日终汇总采集 | 每日16:00 | 采集当日完整数据，进行日度归档 |

### 2.2 渠道采集管理

**一级渠道（官方权威）：**
- 证券交易所官网公告（沪深交易所）
- 上市公司官方公告/通函
- 证监会、银保监会、央行官方发布
- 国家统计局、商务部等宏观数据发布

**二级渠道（主流财经媒体）：**
- 东方财富网、同花顺、雪球
- 财新网、第一财经、证券时报
- 新浪财经、凤凰财经、网易财经
- Bloomberg中国、路透中国

**三级渠道（社交与消息源）：**
- 微博财经话题、Twitter/X财经账号
- 微信公众平台（认证财经号）
- 知乎财经话题、股吧社区
- Telegram财经群组预警

**API数据源：**
- 新闻聚合API（NewsAPI、TinyNewsAPI等）
- 金融数据API（AKShare、TuShare等）
- 社交媒体API（需授权）
- 自建爬虫系统

### 2.3 新闻处理流程

```
原始采集 → 去重清洗 → 格式标准化 → 关键信息提取 → 分类标注 → 质量审核 → 存储输出
```

**步骤1：原始采集**
- 支持多线程并发采集
- 记录采集时间、来源URL、原始HTML/JSON
- 增量采集，只获取新数据

**步骤2：去重清洗**
- 基于标题+内容的SimHash去重
- 过滤广告、推广内容
- 去除无效页面（404、验证码页面等）
- 识别并处理垃圾信息

**步骤3：格式标准化**
- 统一编码（UTF-8）
- 统一时间格式（ISO 8601）
- 提取纯文本内容，去除HTML标签
- 保留原文链接和关键图片描述

**步骤4：关键信息提取**
- 标题清洗和优化
- 核心事件描述（100字内）
- 涉及行业/板块标签
- 涉及公司/股票代码标注
- 原文关键段落保留

**步骤5：分类标注**
- 按新闻类型分类（见分类标准）
- 按行业板块分类
- 按地域分类
- 按影响程度分级（利好/利空/中性）

**步骤6：质量审核**
- 标题完整性检查
- 内容可读性检查
- 关键信息完整性检查
- 分类准确性抽检

**步骤7：存储输出**
- 结构化数据写入数据库
- 原始数据归档存储
- 生成可被下游Agent消费的JSON格式

| 内容完整率 | 完整正文占比 | ≥ 95% |

| 情感判断准确率 | 利好/利空/中性判断 | ≥ 85% |

**数据库表结构（如使用数据库）：**

```sql
CREATE TABLE news_raw (
    news_id VARCHAR(32) PRIMARY KEY,
    title VARCHAR(500) NOT NULL,
    source VARCHAR(100),
    source_url TEXT,
    publish_time DATETIME,
    collect_time DATETIME,
    raw_content LONGTEXT,
    processed_content LONGTEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE news_processed (
    news_id VARCHAR(32) PRIMARY KEY,
    summary TEXT,
    related_stocks JSON,
    category VARCHAR(100),
    sentiment VARCHAR(20),
    impact_level VARCHAR(20),
    FOREIGN KEY (news_id) REFERENCES news_raw(news_id)
);

CREATE INDEX idx_publish_time ON news_processed(publish_time);
CREATE INDEX idx_category ON news_processed(category);
CREATE INDEX idx_sentiment ON news_processed(sentiment);
```

## 四、数据共享机制

### 4.1 下游Agent获取数据方式

**方式1：文件系统共享**
- 处理后的JSON文件存入共享目录
- 股票收集员监控目录变化，实时消费新文件
- 使用文件锁机制避免并发冲突
- 配置示例：`watch_dir = "d:\ai-stocks\data\news\structured"`

**方式2：数据库直接访问**
- 下游Agent拥有只读权限访问news_processed表
- 支持按时间、分类、情感等条件查询
- 提供分页接口，避免大结果集

**方式3：消息队列**
- 使用Redis Pub/Sub或RabbitMQ
- 新闻采集员发布新消息，下游订阅消费
- 支持实时推送，延迟最低

**方式4：API接口**
- 新闻采集员暴露REST API
- 下游Agent通过HTTP请求获取数据
- 支持批量获取和增量获取

### 4.2 数据查询接口规范

```json
// 获取最新新闻
GET /api/news/latest?limit=100&hours=24

// 按分类获取新闻
GET /api/news/category?category=公司财报&limit=50

// 按时间范围获取
GET /api/news/range?start=2026-04-25T00:00:00&end=2026-04-25T23:59:59

// 按股票代码获取相关新闻
GET /api/news/stock?code=600519&days=7

// 获取高影响力利好新闻
GET /api/news/high-impact?sentiment=利好&impact_level=high&limit=20
```

## 五、分类标准体系

### 5.1 新闻类型分类（一级分类）

| 分类代码 | 分类名称 | 说明 | 示例 |
|---------|---------|------|------|
| TYPE_001 | 公司公告 | 上市公司正式公告 | 年报、季报、重大事项 |
| TYPE_002 | 公司财报 | 业绩报告相关 | 营收、利润数据 |
| TYPE_003 | 行业动态 | 行业整体情况 | 政策、趋势、竞争格局 |
| TYPE_004 | 宏观政策 | 宏观经济和政策 | 货币政策、财政政策 |
| TYPE_005 | 市场行情 | 股市整体表现 | 大盘涨跌、成交额 |
| TYPE_006 | 概念热点 | 题材概念炒作 | 元宇宙、AI、新能源 |
| TYPE_007 | 突发事件 | 重大意外事件 | 地震、疫情、事故 |
| TYPE_008 | 监管动态 | 监管政策变化 | 证监会通报、处罚 |
| TYPE_009 | 机构研报 | 券商研究机构报告 | 买入评级、目标价 |
| TYPE_010 | 国际市场 | 海外市场动态 | 美股、港股、外汇 |

### 5.2 行业板块分类（二级分类）

**一级行业：**
- 银行、证券、保险、多元金融
- 白酒、食品饮料、家用电器
- 医药生物、医疗器械
- 房地产、建筑装饰、建筑材料
- 汽车整车、汽车零部件
- 电子元件、半导体、芯片
- 软件开发、IT服务、互联网
- 锂电池、光伏、风电、储能
- 军工、航空航天
- 纺织服装、农林牧渔
- 交通运输、物流
- 传媒、教育、旅游
- 煤炭、石油、天然气
- 钢铁、有色金属、化工

### 5.3 情感分类

| 分类 | 分值范围 | 说明 |
|-----|---------|------|
| 强烈利好 | 0.8 ~ 1.0 | 重大正面消息，可能显著拉升股价 |
| 利好 | 0.5 ~ 0.8 | 正面消息，对股价有正面影响 |
| 中性 | 0.3 ~ 0.5 | 消息面平稳，无明显影响 |
| 利空 | -0.8 ~ -0.3 | 负面消息，对股价有负面影响 |
| 强烈利空 | -1.0 ~ -0.8 | 重大负面消息，可能显著打压股价 |

### 5.4 影响程度分级

| 级别 | 说明 | 适用场景 |
|-----|------|---------|
| high | 重大影响 | 可能影响大盘或板块整体走势 |
| medium | 中等影响 | 主要影响相关个股 |
| low | 轻微影响 | 影响有限或影响滞后 |

### 6.1 功能验收

| 验收项 | 验收条件 | 验证方法 |
|-------|---------|---------|
| 定时采集 | 按设定频率自动执行 | 连续观察24小时，检查日志 |
| 多渠道采集 | 覆盖至少10个主流财经网站 | 抽样检查各渠道数据 |
| 去重功能 | 重复新闻不被重复入库 | 导入1000条已知重复数据，验证去重率 |
| 分类功能 | 一级分类准确率≥95% | 人工标注1000条样本，对比系统分类结果 |
| 情感判断 | 准确率≥85% | 人工标注1000条样本，对比系统判断结果 |
| 股票识别 | 相关股票识别准确率≥90% | 抽样检查新闻中的股票标注 |
| 异常处理 | 网络失败、解析失败时不中断 | 模拟故障场景，验证容错能力 |

## 七、监控与告警

### 7.1 监控指标

- 采集任务执行状态
- 各渠道采集数量统计
- 去重率统计
- 错误率统计
- 处理延迟监控
- 存储空间监控

### 7.2 告警规则

| 告警级别 | 触发条件 | 处理方式 |
|---------|---------|---------|
| 严重 | 采集完全中断超过10分钟 | 立即通知，触发应急 |
| 警告 | 某渠道连续失败超过5次 | 记录日志，更换备选渠道 |
| 提示 | 去重率异常升高（>30%）| 人工检查处理逻辑 |
